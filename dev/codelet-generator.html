The Codelet Generator (genfft analogue) — Knowledge Base ​

This is the reference knowledge base for PureFFT's Julia-native FFT codelet generator — the analogue of FFTW's genfft. It captures the literature on how genfft works, the SIMD-vectorization theory, and how both map onto PureFFT's existing infrastructure, so the design and implementation can refer to it without re-deriving. The architecture + PoC plan that consumes this KB lives in the plan file (docs/superpowers/plans/… / the approved plan); this page is the background, not the plan.

Constraint: we cannot read FFTW's source. Everything here is from the published papers, the Burrus textbook, and FFTW.jl's bindings — never FFTW's C/OCaml source. Sources are listed at the bottom.

Why a codelet generator (the Julia-only advantage) ​

FFTW is fast because its small fixed-size FFT kernels ("codelets") are generated by a program (genfft) that applies FFT-specific optimizations a general compiler does not. But genfft runs once, at FFTW build time, and the codelets are compiled into the library; RustFFT similarly fixes its kernels at compile time. Neither can synthesize a new optimized kernel for a size it did not anticipate — they fall back to slower general routines.

Julia can: @generated functions (and runtime eval) specialize code per size, on first use. PureFFT already exploits a limited form of this. A full generator would give optimized codelets for any size on demand — a capability that is structurally impossible for an ahead-of-time C/Rust library, and the clearest reason a from-scratch Julia FFT can be more than "FFTW re-implemented."

genfft — the four phases ​

genfft (FFTW's generator, written in OCaml) compiles a size-n DFT into a straight-line codelet in four phases:

1. Creation ​

Build a directed acyclic graph (DAG) of complex operations (+, −, ×constant) by evaluating a DFT algorithm symbolically — "expressed in a straightforward math-like notation, using complex numbers, with no attempt at optimization." The algorithm is chosen per size:

  • Cooley–Tukey (n = n₁·n₂, the general recursive factorization),

  • split-radix (the lowest known operation count for powers of two),

  • prime-factor / Good–Thomas (coprime factors, no twiddles between stages),

  • Rader (a prime p as a length-p−1 cyclic convolution).

The output is a linear network where edges carry multiplicative constants and vertices perform additions.

2. Simplification ​

Local rewrite rules applied to every DAG node, plus global passes. This is where genfft earns most of its advantage over naively-emitted code:

  • Algebraic simplification / constant folding: eliminate ×0, ×1, ×−1, and the special rotations ×±i (which are lane swaps + sign flips, not multiplies); fold constant × constant.

  • Sign propagation: "make all the numeric constants positive and propagate the minus sign along the edges of the DAG" — turning additions into subtractions (or vice versa) elsewhere. Reported 10–15% speedup. (Fewer distinct constants → more CSE hits and simpler immediates.)

  • Common-subexpression elimination (CSE): hash-cons identical subgraphs so each is computed once. This is the single largest op-count reducer — and the one a modern backend compiler is also likely to do (see "the make-or-break question").

  • Network transposition: reverse the direction of every edge in the DAG, run CSE again, then transpose back. By the network-transposition theorem the transposed graph computes the transposed linear map; CSE on it exposes common subexpressions that a single forward pass misses. A genfft-specific trick.

  • Real-data specialization: automatically prune the complex DAG to derive real-input (r2c / r2r) algorithms at near-minimal operation count.

3. Scheduling ​

Produce a topological sort of the simplified DAG — an order of operations — chosen so the backend compiler can register-allocate well. genfft's scheduler is cache-oblivious, treating the register file as a cache: it recursively partitions the DAG to keep the working set (live values) small, which bounds register pressure / spills. The theory connects to cache-oblivious algorithms and the red–blue pebble game. "One consequence of this scheduler is that FFTW's machine-independent codelets are no slower than machine-specific codelets generated by SPIRAL."

4. Unparse ​

Emit the scheduled DAG as straight-line code (C in stock genfft; assembly optionally). Two flavors:

  • No-twiddle codelet: a plain size-n DFT (used as a leaf / base case).

  • Twiddle codelet: inputs/outputs are pre/post-multiplied by twiddle factors, so the codelet drops directly into a Cooley–Tukey step. Codelet sizes are typically n ≤ 64. The planner composes these codelets (timing combinations) into a plan for the full transform.

SIMD: vectorize straight-line blocks, not loops ​

A general loop-vectorizing compiler cannot vectorize FFTW: the strides are runtime-determined and most codelets have no loops (they are straight-line blocks). FFTW instead vectorizes the straight-line code itself, with SIMD lanes holding independent sub-transforms (the "vector rank" / batch dimension). The data layout is split (separate real/imaginary streams) so the arithmetic is plain FMA chains with no intra-transform shuffles or transposes.

This is the key fact for PureFFT: it is the batch-vectorization axis, and it is transpose-free. It contrasts with column-packing (packing several columns of one transform into a SIMD register), which incurs intra-transform transpose shuffles. PureFFT's measured non-pow2 floors (e.g. 2^a·5³) were root-caused to exactly this — FFTW vectorizes across the batch, PureFFT column-packs. The canonical reference for how to vectorize straight-line FFT code is Franchetti et al. (see sources).

How this maps onto PureFFT (existing seeds + the gap) ​

PureFFT is not greenfield here:

  • src/codelets.jl is a working genfft seed. Its four generators (_gen_dft!, _gen_dft_mixed!, _gen_dft_soa!, _gen_dft_soa_mixed!) build straight-line DFT code at @generated-dispatch time with baked twiddle constants and Cooley–Tukey mixed-radix recursion, for any size — a complete Creation phase plus a sliver of Simplification (it already elides ×1 and ×±i). It powers CodeletPlan, FourStepCodeletPlan, and RecursiveMixedRadixPlan.

  • The batch-vectorization axis already exists in seed form. _dft_codelet_soa_batched! / _dft_codelet_soa_batched_tw! emit a Vec{W,T} body vectorizing across W independent transforms (split re/im, no intra-transform shuffles) — FFTW's transpose-free strategy. FourStepCodeletPlan drives two such passes for a four-step n = n₁·n₂.

  • What's missing is exactly genfft's optimization layer: there is no explicit IR, no CSE, no sign-propagation, no scheduling, no network-transposition. The generators emit the naive DAG and rely on LLVM.

  • The src/avxradix/ SIMD layer is the hand-written, per-size "ad-hoc" part (~50 functions across two register widths; a new radix touches ~6 places). avx_colbf_prime (a @generated width-generic prime butterfly) is its only generative element. Systematizing this layer is the generator's long-term goal.

So the generator is built by inserting an explicit IR + a few optimization passes between the existing frontends and backends, and promoting the batched backend to first-class.

The make-or-break question: does this beat modern LLVM? ​

genfft's advantage was measured against 1990s C compilers. Modern LLVM already performs aggressive CSE, instruction scheduling, and register allocation. PureFFT's own julia-sched-mwe/ reproducer found that LLVM compiles our radix-9/12 DAGs to asm at least as fast as Rust's — i.e. the classic genfft "schedules better than the compiler" edge may be narrow against LLVM 18+.

Therefore the implementation must, before building any compiler, measure (via @code_native instruction-count diffs on a real codelet) whether each genfft pass actually changes what LLVM emits. A pass LLVM already performs is deleted, not kept "for completeness." The verified-value layer is batch-vectorization (which mostly already exists and addresses a measured floor); the scalar CSE/schedule passes are the high-risk, possibly-redundant layer and are gated by measurement.

Trim-safety and the generate-time / runtime split ​

The symbolic DAG/CSE/scheduling work uses dynamic structures (Vector{Any}-style) that must never appear in the runtime hot path (PureFFT targets --trim / static compilation; verified via TrimCheck and StrictMode). The resolution mirrors genfft: the generator runs entirely at generate time (inside @generated bodies or a precompile build step) and emits only concrete, straight-line code. The runtime is the emitted kernel — trim-safe by construction.
